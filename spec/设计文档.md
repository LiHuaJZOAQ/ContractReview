### 1.3 技术栈选型理由

| 技术 | 选型理由 |
|------|----------|
| **Vue 3 + Element Plus** | 响应式前端框架，组合式 API 便于组织复杂逻辑；Element Plus 提供成熟的表格、表单、进度条等组件，适合企业级管理界面快速开发 |
| **Axios** | 基于 Promise 的 HTTP 客户端，支持请求/响应拦截器，便于统一添加 JWT 令牌和处理错误 |
| **Spring Boot 3** | 成熟的 Java 后端框架，提供自动配置、依赖注入、声明式事务等特性；Spring Boot 3 最低要求 JDK 17，性能更优，安全性更强 |
| **Spring Security** | 与 Spring Boot 深度集成，提供 JWT 认证、角色授权、CSRF 防护等开箱即用功能 |
| **Spring AI** | 官方 AI 框架，提供统一的 LLM 调用抽象、Prompt 模板管理、向量数据库集成、OutputParser 等，避免锁定特定 AI 供应商 |
| **RabbitMQ** | 成熟的消息中间件，支持死信队列 (DLX)、延迟重试、消息确认，适合长耗时任务的异步解耦与削峰填谷 |
| **MySQL 8.0** | 关系型数据库，支持 JSON 数据类型（存储脱敏预览和报告 JSON）、窗口函数、CTE，满足业务数据一致性要求 |
| **Redis** | 高性能内存缓存，原生支持 Lua 脚本（用于滑动窗口限流）、DECR/INCR 原子操作（用于配额扣减）、键过期（TTL 缓存），适合高频低延迟场景 |
| **Chroma** | 轻量级向量数据库，支持本地部署，无需额外 GPU 资源；Python 与 Java 均有 HTTP 客户端，适合中小规模向量检索场景 |
| **MinIO / 阿里云 OSS** | 对象存储，用于持久化合同文件与 PDF 报告。MinIO 兼容 S3 API 适合私有化部署，OSS 适合云上部署 |
| **Apache PDFBox** | Apache 基金会的 PDF 处理库，支持文本提取、文档创建，无第三方依赖，适合服务端 PDF 解析 |
| **Apache POI** | Apache 基金会的 Office 文档处理库，支持 Word (docx) 解析与文本提取，是 Java 生态的标准选择 |

#### 2.1.1 PDF 解析流程 (Apache PDFBox)


关键实现细节：

- 使用 `PDDocument.load(inputStream)` 加载文档
- 使用 `PDFTextStripper` 逐页提取文本，设置 `setSortByPosition(true)` 保持阅读顺序
- 提取过程中记录文本在原始文档中的位置偏移，用于脱敏预览时高亮标注
- 解析超时控制在 30s 以内（SRS 要求 PARSING ≤ 30s）
- 解析完成后立即关闭 `PDDocument` 释放资源

关键实现细节：

- 遍历 `XWPFParagraph` 列表，通过 `getText()` 提取段落文本

- 解析超时控制在 30s 以内

**管道设计模式：**

每个脱敏阶段实现 `Desensitizer` 接口：

```java
public interface Desensitizer {
    DesensitizeResult process(String text, DesensitizeContext context);
}
```

- 各阶段按优先级顺序执行，前一阶段的输出作为后一阶段的输入
- 每个阶段输出 `DesensitizeResult`，包含脱敏后的文本和该阶段发现的敏感信息列表
- 所有阶段的结果汇总到最终的脱敏映射表，用于前端预览高亮
- P1 NLP 增强阶段可通过配置开关启用/禁用，默认仅在 P1 优先级需求时开启

#### 2.1.4 脱敏预览→确认提交流程

```
 ┌──────────┐    ┌──────────────┐    ┌──────────┐    ┌────────────┐    ┌──────────┐
 │ POST     │    │ 上传 → 解析   │    │ 返回     │    │ POST       │    │ 提交 →   │
 │ /upload  │───▶│ → 脱敏       │───▶│ 脱敏预览  │───▶│ /{taskId}/ │───▶│ 扣配额   │
 │          │    │ 创建 PENDING │    │ + taskId  │    │ submit     │    │ → 投递MQ │
 │          │    │ 状态任务     │    │ (不扣配额) │    │            │    │ → PARSING│
 └──────────┘    └──────────────┘    └──────────┘    └────────────┘    └──────────┘
                                                                              │
                                                                        ┌─────▼─────┐
                                                                        │ 1h 后未   │
                                                                        │ submit 则 │
                                                                        │ 自动清理   │
                                                                        │ PENDING   │
                                                                        │ 任务+文件  │
                                                                        └───────────┘
```

**流程说明：**

1. 用户上传文件 → 服务端校验格式和大小 → 解析并脱敏 → 创建 `PENDING` 状态任务 → 返回 `{taskId, previewText}`
2. 用户预览脱敏结果 → 确认后调用 `submit` → 服务端扣减配额（Redis DECR）→ 投递 MQ 消息 → 修改状态为 `PARSING`
3. 如果用户 1 小时内未确认提交，定时任务自动清理 `PENDING` 任务及临时文件

### 2.2 RAG 法条检索模块

#### 2.2.1 Chunking 策略

```
 ┌─────────────────────────────────────────────────────────────────────────┐
 │                        Chunking 分块处理器                              │
 │                                                                         │
 │  输入: 合同全文文本                                                     │
 │                                                                         │
 │  步骤 1: 按条款分割                                                     │
 │  ┌──────────────────────────────────────────────────────────────────┐   │
 │  │ 正则模式: ^(第[一二三四五六七八九十百千]+条|第\d+条|Art\.\d+)    │   │
 │  │ 匹配到条款标题时切分，每个条款为一个 Chunk                       │   │
 │  └──────────────────────────────────────────────────────────────────┘   │
 │                                                                         │
 │  步骤 2: 段落分割 (无条款结构时)                                        │
 │  ┌──────────────────────────────────────────────────────────────────┐   │
 │  │ 按连续空行(\n\n+)分割, 每个段落为一个 Chunk                      │   │
 │  └──────────────────────────────────────────────────────────────────┘   │
 │                                                                         │
 │  步骤 3: 长度调整                                                      │
 │  ┌──────────────────────────────────────────────────────────────────┐   │
 │  │ 若 Chunk > 800 tokens → 递归切分至 ≤800 tokens                  │   │
 │  │ 若 Chunk < 50 tokens  → 与相邻 Chunk 合并                      │   │
 │  │ 保留 Chunk 索引 (clause_index) 用于最终报告定位                  │   │
 │  └──────────────────────────────────────────────────────────────────┘   │
 │                                                                         │
 │  输出: List<Chunk> { index, content, startPos, endPos }               │
 └─────────────────────────────────────────────────────────────────────────┘
```

**上下文关联保留策略：**

- 每个 Chunk 携带前一个条款的标题作为上下文前缀（如 "（接上一条：第十三条 违约责任）"）
- 当 Chunk 因长度调整被拆分时，拆分后的子 Chunk 共享同一个 `clause_index`
- 在 Embedding 时，将条款标题与内容拼接后向量化，提升检索准确性

#### 2.2.2 Embedding 模型选型建议

| 模型 | 维度 | 适用场景 | 推荐理由 |
|------|------|----------|----------|
| **text-embedding-3-small** (OpenAI) | 1536 | 通用 | 性价比高，适合各类法律文本 |
| **text-embedding-3-large** (OpenAI) | 3072 | 高精度 | 精度最高，成本和延迟略高 |
| **BAAI/bge-large-zh-v1.5** (开源) | 1024 | 中文优化 | 本地部署，中文法律文本效果好 |
| **通义千问 text-embedding-v2** | 1536 | 中文优化 | 阿里云生态，中文理解强 |

**推荐方案：** 生产环境使用 `text-embedding-3-small` 作为主要模型，开源阶段使用 `BAAI/bge-large-zh-v1.5` 进行本地开发和测试。配置应支持动态切换 Embedding 模型（通过 Spring AI 的 `EmbeddingModel` 抽象）。

#### 2.3.5 Map-Reduce 详细设计

```
 ┌─────────────────────────────────────────────────────────────────────────┐
 │                      Map-Reduce 并发引擎                                │
 │                                                                         │
 │  ┌──────────────────────────────────────────────────────────────────┐   │
 │  │  Map 阶段                                                        │   │
 │  │                                                                  │   │
 │  │  ThreadPool:                                                        │   │
 │  │  - corePoolSize = 10                                             │   │
 │  │  - maxPoolSize = 20                                              │   │
 │  │  - queueCapacity = 50                                            │   │
 │  │  - 使用有界队列 + CallerRunsPolicy                                │   │
 │  │                                                                  │   │
 │  │  Semaphore: maxConcurrent = 10 (可通过配置调整)                   │   │
 │  │                                                                  │   │
 │  │  每个 Chunk 的处理流程:                                           │   │
 │  │  1. semaphore.acquire()  ← 阻塞直到获得信号量                    │   │
 │  │  2. chunk → Embedding → Chroma 检索 → 获取相关法条               │   │
 │  │  3. 组装 Agent B Prompt → 调用 LLM                              │   │
 │  │  4. 解析 Agent B 输出 → 提取风险点列表                           │   │
 │  │  5. 风险点写入 risk_item 表 (批量)                               │   │
 │  │  6. reviewed_chunks + 1, progress 更新                           │   │
 │  │  7. SSE 推送进度更新                                              │   │
 │  │  8. semaphore.release()                                          │   │
 │  └──────────────────────────────────────────────────────────────────┘   │
 │                                                                         │
 │  ┌──────────────────────────────────────────────────────────────────┐   │
 │  │  Reduce 阶段                                                      │   │
 │  │                                                                  │   │
 │  │  等待所有 Agent B 完成后 (CompletableFuture.allOf)               │   │
 │  │                                                                  │   │
 │  │  1. 从 risk_item 表查询当前任务所有风险点                          │   │
 │  │  2. 按风险等级分组 (HIGH/MEDIUM/LOW)                              │   │
 │  │  3. 组装 Agent C Prompt → 调用 LLM                              │   │
 │  │  4. 解析 Agent C 输出 → 生成 JSON 报告                           │   │
 │  │  5. 写入 review_report 表                                        │   │
 │  │  6. 更新任务状态为 SUCCESS, progress = 100                       │   │
 │  │  7. SSE 推送 complete 事件                                       │   │
 │  └──────────────────────────────────────────────────────────────────┘   │
 └─────────────────────────────────────────────────────────────────────────┘
```

#### 2.4.2 消费者设计

```java
@Component
@Slf4j
public class ReviewTaskConsumer {

    @Autowired
    private ReviewTaskService reviewTaskService;

    @RabbitListener(queues = "contract.review.queue", ackMode = "MANUAL")
    public void handleReviewTask(ReviewTaskMessage message, Channel channel,
                                  @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag) {
        try {
            // 1. 更新任务状态为 PARSING
            reviewTaskService.startParsing(message.getTaskId());

            // 2. 执行完整审查工作流 (Agent A → Chunking → RAG + Agent B → Agent C)
            reviewTaskService.executeReviewWorkflow(message.getTaskId());

            // 3. 手动确认消息
            channel.basicAck(deliveryTag, false);

        } catch (Exception e) {
            log.error("审查任务处理失败, taskId={}", message.getTaskId(), e);

            // 判断重试次数
            MessageProperties props = message.getProperties();
            long retryCount = Optional.ofNullable(props.getHeader("x-retry-count"))
                .map(h -> (Long) h).orElse(0L);

            if (retryCount < 3) {
                // 重新投递并设置延迟
                // (使用 Spring Retry 机制处理)
                throw new RetryableException(e);
            } else {
                // 超过重试次数, 投递 DLQ
                channel.basicNack(deliveryTag, false, false);
                reviewTaskService.handleTaskFailed(message.getTaskId(),
                    "审查任务失败: " + e.getMessage());
            }
        }
    }
}
```

#### 2.4.3 SSE 推送实现

```java
@Service
public class SseService {

    /**
     * taskId → SseEmitter 映射
     * 使用 ConcurrentHashMap 保证线程安全
     */
    private final ConcurrentHashMap<String, SseEmitter> emitters = new ConcurrentHashMap<>();

    private static final long SSE_TIMEOUT = 5 * 60 * 1000L; // 5分钟

    /**
     * 创建 SSE 连接
     */
    public SseEmitter createEmitter(String taskId) {
        // 超时时间设为略大于总超时时间 (5min)
        SseEmitter emitter = new SseEmitter(SSE_TIMEOUT);

        emitters.put(taskId, emitter);

        emitter.onCompletion(() -> emitters.remove(taskId));
        emitter.onTimeout(() -> emitters.remove(taskId));
        emitter.onError(e -> emitters.remove(taskId));

        return emitter;
    }

    /**
     * 推送进度事件
     */
    public void sendProgress(String taskId, String stage, int progress, String message) {
        SseEmitter emitter = emitters.get(taskId);
        if (emitter == null) return;

        try {
            Map<String, Object> data = new HashMap<>();
            data.put("event", "progress");
            data.put("stage", stage);       // parsing | retrieving | reviewing | summarizing
            data.put("progress", progress);  // 0-100
            data.put("message", message);
            data.put("timestamp", System.currentTimeMillis());

            emitter.send(SseEmitter.event()
                .name("progress")
                .data(data, MediaType.APPLICATION_JSON));
        } catch (IOException e) {
            emitters.remove(taskId);
        }
    }

    /**
     * 推送完成事件
     */
    public void sendComplete(String taskId, UUID reportId) {
        SseEmitter emitter = emitters.get(taskId);
        if (emitter == null) return;

        try {
            Map<String, Object> data = new HashMap<>();
            data.put("event", "complete");
            data.put("taskId", taskId);
            data.put("reportId", reportId.toString());
            data.put("timestamp", System.currentTimeMillis());

            emitter.send(SseEmitter.event()
                .name("complete")
                .data(data, MediaType.APPLICATION_JSON));
            emitter.complete();
        } catch (IOException e) {
            emitters.remove(taskId);
        }
    }

    /**
     * 推送错误事件
     */
    public void sendError(String taskId, String errorCode, String errorMessage) {
        SseEmitter emitter = emitters.get(taskId);
        if (emitter == null) return;

        try {
            Map<String, Object> data = new HashMap<>();
            data.put("event", "error");
            data.put("code", errorCode);
            data.put("message", errorMessage);
            data.put("timestamp", System.currentTimeMillis());

            emitter.send(SseEmitter.event()
                .name("error")
                .data(data, MediaType.APPLICATION_JSON));
            emitter.complete();
        } catch (IOException e) {
            emitters.remove(taskId);
        }
    }
}
```

**SSE 事件类型汇总：**

| 事件名 | 触发时机 | 数据字段 |
|--------|----------|----------|
| `progress` | 阶段切换或进度更新 | `{event, stage, progress, message, timestamp}` |
| `complete` | 审查完成 | `{event, taskId, reportId, timestamp}` |
| `error` | 审查失败 | `{event, code, message, timestamp}` |

```
 │   └──────────┘

 │  回退路径:
 │  FAILED → PENDING (手动重试, 重新开始整个流程)
```

**各阶段处理逻辑描述：**

| 状态 | 处理逻辑 | 超时阈值 | SSE 事件 |
|------|----------|----------|----------|
| `PENDING` | 文件已上传并脱敏，等待用户确认提交。1h 后未提交自动清理 | — | — |
| `PARSING` | 读取脱敏文件内容，执行 Chunking 分块。完成后状态→`RETRIEVING` | 30s | `progress {stage: "parsing", progress: 0-20}` |
| `RETRIEVING` | 对所有 Chunk 执行向量化 + Chroma 检索，必要时 fallback 网络搜索。完成后状态→`REVIEWING` | 30s | `progress {stage: "retrieving", progress: 20-30}` |
| `REVIEWING` | Agent B 并发扫描每个 Chunk（Map 阶段）。每个 Chunk 完成后更新进度。全部完成后状态→`SUMMARIZING` | 120s | `progress {stage: "reviewing", progress: 30-80}` |
| `SUMMARIZING` | Agent C 汇总风险点生成 JSON 报告。完成后状态→`SUCCESS` | 30s | `progress {stage: "summarizing", progress: 80-100}` → `complete` |
| `SUCCESS` | 审查完成，报告可查询。前端收到 `complete` 事件后跳转报告页 | — | — |
| `FAILED` | 任意阶段发生异常。记录 `error_msg`，前端收到 `error` 事件。用户可手动重试 | — | `error {code, message}` |

**状态转换枚举定义：**

```java
public enum TaskStatus {
    PENDING,
    PARSING,
    RETRIEVING,
    REVIEWING,
    SUMMARIZING,
    SUCCESS,
    FAILED;

    private static final Set<TaskStatus> VALID_TRANSITIONS = new HashSet<>();

    static {
        // PENDING → PARSING (提交确认)
        // PENDING → DELETED (1h 自动清理, 逻辑删除)
        // PARSING → RETRIEVING (解析完成)
        // PARSING → FAILED (解析失败)
        // RETRIEVING → REVIEWING (检索完成)
        // RETRIEVING → FAILED (检索失败)
        // REVIEWING → SUMMARIZING (扫描完成)
        // REVIEWING → FAILED (扫描失败)
        // SUMMARIZING → SUCCESS (报告生成完成)
        // SUMMARIZING → FAILED (报告生成失败)
        // FAILED → PENDING (手动重试)
    }

    public boolean canTransitionTo(TaskStatus target) {
        // 实现状态转换校验逻辑
        switch (this) {
            case PENDING:
                return target == PARSING || target == FAILED;
            case PARSING:
                return target == RETRIEVING || target == FAILED;
            case RETRIEVING:
                return target == REVIEWING || target == FAILED;
            case REVIEWING:
                return target == SUMMARIZING || target == FAILED;
            case SUMMARIZING:
                return target == SUCCESS || target == FAILED;
            case FAILED:
                return target == PENDING;
            case SUCCESS:
                return false; // 终态
            default:
                return false;
        }
    }
}
```

```sql

-- 1. 按用户查询任务列表 (分页)
-- ============================================================
SELECT
    rt.id,
    rt.task_id,
    rt.file_name,
    rt.file_size,
    rt.contract_type,

    rt.progress,
    rt.total_chunks,
    rt.reviewed_chunks,
    rt.created_at,
    rt.completed_at

WHERE rt.user_id = ?
ORDER BY rt.created_at DESC
LIMIT ? OFFSET ?;

-- ============================================================
-- 2. 按任务查询所有风险项
-- ============================================================
SELECT
    ri.id,
    ri.clause_index,
    ri.clause_content,
    ri.risk_level,
    ri.risk_type,
    ri.description,
    ri.suggestion,
    ri.related_laws,
    ri.created_at
FROM risk_item ri
WHERE ri.task_id = ?
ORDER BY ri.risk_level ASC, ri.clause_index ASC;
-- risk_level 排序: HIGH < MEDIUM < LOW (ENUM 按定义顺序)

-- ============================================================
-- 3. 统计风险等级分布
-- ============================================================
SELECT
    ri.risk_level,
    COUNT(*) AS count
FROM risk_item ri
WHERE ri.task_id = ?
GROUP BY ri.risk_level
ORDER BY ri.risk_level;

-- ============================================================
-- 4. 获取审查报告 (含统计)
-- ============================================================
SELECT
    rr.summary,
    rr.risk_count_high,
    rr.risk_count_medium,
    rr.risk_count_low,
    rr.report_json,
    rr.pdf_url,
    rr.created_at
FROM review_report rr
WHERE rr.task_id = ?;

-- ============================================================
-- 5. 查询用户当前配额
-- ============================================================
SELECT
    u.review_quota,
    u.version
FROM user u
WHERE u.id = ?;

-- ============================================================
-- 6. 配额扣减 (乐观锁)
-- ============================================================
UPDATE user u
SET
    u.review_quota = u.review_quota - 1,
    u.version = u.version + 1
WHERE u.id = ?
  AND u.review_quota > 0
  AND u.version = ?;

-- ============================================================
-- 7. 配额回滚 (任务失败时)
-- ============================================================
UPDATE user u
SET
    u.review_quota = u.review_quota + 1,
    u.version = u.version + 1
WHERE u.id = ?
  AND u.version = ?;

-- ============================================================
-- 8. 查找待清理的 PENDING 任务 (超过1小时)
-- ============================================================
SELECT
    rt.id,
    rt.task_id,
    rt.file_url
FROM review_task rt
WHERE rt.status = 'PENDING'
  AND rt.created_at < DATE_SUB(NOW(), INTERVAL 1 HOUR);

-- ============================================================
-- 9. 法条管理: 按法规名查询所有条款
-- ============================================================
SELECT
    lr.article_num,
    lr.content,
    lr.chapter
FROM law_regulation lr
WHERE lr.law_name = ?
ORDER BY lr.article_num;

```

```
 │                          SSE 重连机制                                   │
 │                                                                         │
 │  前端 (EventSource)                         后端 (SseEmitter)           │
 │  ┌──────────────────────┐                  ┌──────────────────────┐    │
 │  │ 1. 建立连接           │──── SSE GET ────▶│ 2. 创建 SseEmitter   │    │

 │  └──────────┬───────────┘                                              │

 │  ┌──────────▼───────────┐                                              │

 │  │    EventSource 自带   │    (带 Last-Event-   │                       │

 │  ├── 首次重连延迟: 3s (EventSource 默认)
 │  ├── 最大重连次数: 5 次
```

**前端 EventSource 封装示例：**

```javascript
class ReviewSSEClient {
    constructor(taskId, token) {
        this.taskId = taskId;
        this.token = token;
        this.eventSource = null;
        this.retryCount = 0;
        this.maxRetries = 5;
    }

    connect() {
        const url = `/api/v1/contract/${this.taskId}/progress?token=${this.token}`;
        this.eventSource = new EventSource(url);

        this.eventSource.addEventListener('progress', (event) => {
            const data = JSON.parse(event.data);
            // 更新进度条和状态文字
            this.onProgress(data);
            this.retryCount = 0; // 收到正常事件后重置重试计数
        });

        this.eventSource.addEventListener('complete', (event) => {
            const data = JSON.parse(event.data);
            this.onComplete(data);
            this.eventSource.close();
        });

        this.eventSource.addEventListener('error', (event) => {
            const data = JSON.parse(event.data);
            this.onError(data);
            this.eventSource.close();
        });

        this.eventSource.onerror = () => {
            if (this.retryCount < this.maxRetries) {
                this.retryCount++;
                // EventSource 会自动重连, 无需手动处理
            } else {
                this.eventSource.close();
                this.onMaxRetriesExceeded();
            }
        };
    }

    disconnect() {
        if (this.eventSource) {
            this.eventSource.close();
            this.eventSource = null;
        }
    }
}
```

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) {
        String token = extractToken(request);
        if (token != null) {
            try {
                Claims claims = Jwts.parser()
                    .verifyWith(signingKey)
                    .build()
                    .parseSignedClaims(token)
                    .getPayload();

                // 检查是否在黑名单中 (用户登出时加入)
                if (redisTemplate.hasKey("jwt:blacklist:" + claims.getId())) {
                    throw new JwtException("Token has been revoked");
                }

                // 设置 SecurityContext
                UsernamePasswordAuthenticationToken auth =
                    new UsernamePasswordAuthenticationToken(
                        claims.getSubject(), null, authorities);
                SecurityContextHolder.getContext().setAuthentication(auth);

            } catch (JwtException e) {
                // 令牌无效, 返回 401
                response.setStatus(HttpStatus.UNAUTHORIZED.value());
                return;
            }
        }
        filterChain.doFilter(request, response);
    }

    private String extractToken(HttpServletRequest request) {
        String header = request.getHeader("Authorization");
        if (header != null && header.startsWith("Bearer ")) {
            return header.substring(7);
        }
        // SSE 连接可通过 query 参数传递 token
        return request.getParameter("token");
    }
}
```

### 5.2 Redis 滑动窗口限流 Lua 脚本

#### 5.2.1 限流配置

```
限流目标: 每个用户每 60 秒最多 30 次 API 请求
限流粒度: 按用户 ID + API 路径
窗口大小: 60 秒
最大请求数: 30
```

#### 5.2.2 Lua 脚本

```lua
-- 滑动窗口限流 Lua 脚本
-- KEYS[1]: rate_limit:{userId}:{path}   (Redis key)
-- ARGV[1]: 窗口大小 (毫秒), 例如 60000
-- ARGV[2]: 最大请求数, 例如 30
-- ARGV[3]: 当前时间戳 (毫秒)
-- ARGV[4]: 本次请求的权重 (默认 1)

-- 1. 清理窗口外的旧记录
redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, ARGV[3] - ARGV[1])

-- 2. 统计当前窗口内的请求数
local current = redis.call('ZCARD', KEYS[1])

-- 3. 检查是否超过阈值
if current >= tonumber(ARGV[2]) then
    -- 获取窗口内最早的请求时间, 计算重试等待时间
    local earliest = redis.call('ZRANGE', KEYS[1], 0, 0, 'WITHSCORES')
    local retryAfter = (tonumber(ARGV[1]) - (ARGV[3] - earliest[2])) / 1000
    return {0, math.ceil(retryAfter)}  -- 0 = 拒绝, retryAfter = 建议等待秒数
end

-- 4. 记录本次请求
redis.call('ZADD', KEYS[1], ARGV[3], ARGV[3] .. ':' .. math.random())

-- 5. 设置过期时间 (防止内存泄漏)
redis.call('EXPIRE', KEYS[1], ARGV[1] / 1000 + 1)

return {1, 0}  -- 1 = 允许, 0 = 无需等待
```

#### 5.2.3 Java 调用

```java
@Service
public class RateLimiter {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final String LUA_SCRIPT = "..."; // 上述 Lua 脚本

    public RateLimitResult allowRequest(Long userId, String path) {
        String key = "rate_limit:" + userId + ":" + path;
        long now = System.currentTimeMillis();

        List<Long> result = redisTemplate.execute(
            new DefaultRedisScript<>(LUA_SCRIPT, Long.class),
            List.of(key),
            String.valueOf(60000),  // 窗口大小 60s
            String.valueOf(30),     // 最大 30 次
            String.valueOf(now),
            "1"
        );

        boolean allowed = result.get(0) == 1;
        int retryAfter = result.get(1).intValue();

        return new RateLimitResult(allowed, retryAfter);
    }
}
```

### 5.3 配额扣减与回滚

```
 ┌──────────┐          ┌────────────┐          ┌──────────┐
 │ 用户提交   │─────────▶│ Redis DECR │─────────▶│ 检查结果  │
 │ 审查任务   │          │ quota:     │          │          │
 │           │          │ user:{id}  │          │          │
 └──────────┘          └──────┬─────┘          └─────┬────┘
                              │                      │
                              │               ┌──────▼──────┐
                              │               │ DECR 结果 < 0│
                              │               │ (配额不足)   │
                              │               └──────┬──────┘
                              │                      │
                              │               ┌──────▼──────┐
                              │               │ INCR 回滚    │
                              │               │ 返回配额不足  │
                              │               │ 错误码 1004  │
                              │               └─────────────┘
                              │
                              │          ┌────────────┐
                              │─────────▶│ 投递 MQ    │
                              │          │ 开始审查    │
                              │          └────────────┘
                              │
                         ┌────▼────┐
                         │ 任务成功  │
                         │ 或失败    │
                         └────┬────┘
                              │
                    ┌─────────▼──────────┐
                    │ 任务失败:           │
                    │ Redis INCR 回滚配额  │
                    │ 数据库乐观锁回滚     │
                    └────────────────────┘

  数据库乐观锁回滚:
  ┌──────────────────────────────────────────────────────────────────────┐
  │ // 1. 提交时扣减数据库配额 (乐观锁)                                   │
  │ int rows = jdbcTemplate.update(                                      │
  │     "UPDATE user SET review_quota = review_quota - 1, version =      │
  │      version + 1 WHERE id = ? AND review_quota > 0 AND version = ?",│
  │     userId, currentVersion);                                         │
  │                                                                      │
  │ // 2. 任务失败时回滚 (乐观锁)                                         │
  │ int rows = jdbcTemplate.update(                                      │
  │     "UPDATE user SET review_quota = review_quota + 1, version =      │
  │      version + 1 WHERE id = ? AND version = ?",                     │
  │     userId, currentVersion);                                         │
  └──────────────────────────────────────────────────────────────────────┘
```

**配额扣减时序：**

1. `submit` 接口被调用
2. Redis `DECR quota:user:{userId}` 预扣配额
3. 如果 DECR 结果 < 0，执行 `INCR` 回滚并返回配额不足错误
4. DECR 成功 → 写入 `review_task` 记录（状态 `PARSING`）→ 投递 MQ
5. 异步任务完成（成功/失败）后，数据库乐观锁扣减/回滚永久配额
6. Redis 配额仅用于快速预检，数据库配额为最终依据

```
 │  │    权限: 仅服务进程可读写 (600)                                      │
 │  └─────────────────┬────────────────────┘                               │
 │                    │                                                     │

 │  └─────────────────┬────────────────────┘                               │
 │                    │                                                     │
 │                    ▼                                                     │
 │  ┌──────────────────────────────────────┐                               │

 │  │    · 管道式正则+NLP脱敏              │                               │
 │  │    · 生成脱敏后文本                   │                               │
 │  │    · 记录脱敏映射（用于前端高亮）      │                               │
 │  └─────────────────┬────────────────────┘                               │
 │                    │                                                     │
 │                    ├────────────────────────────┐                       │
 │                    ▼                            ▼                       │
 │  ┌────────────────────────────┐  ┌────────────────────────────┐        │
 │  │ 4a. 原始文件处理            │  │ 4b. 脱敏文件持久化          │        │
 │  │    · 脱敏预览返回后立即删除   │  │    · 上传至 MinIO/OSS      │        │
 │  │    · 原始文本不在内存中停留   │  │    · 路径: desensitized/   │        │
 │  │    · 不写入数据库或持久化存储  │  │      {taskId}.txt/pdf     │        │
 │  └────────────────────────────┘  │    · 保留脱敏映射 JSON      │        │

 │                                   │      高亮展示, 不持久化    │        │
 │                                   └────────────────────────────┘        │
 │                                                                         │
 │  ┌──────────────────────────────────────────────────────────────────┐   │
 │  │ 5. 临时文件清理                                                     │   │
 │  │    · PENDING 任务: 1h 后自动清理临时目录                           │   │
 │  │    · 已完成任务: 保留脱敏文件, 删除原始文件临时副本                  │   │
 │  │    · 定时任务: 每小时扫描一次 /tmp/contract_review/ 清理过期文件   │   │
 │  └──────────────────────────────────────────────────────────────────┘   │
 └─────────────────────────────────────────────────────────────────────────┘
```

**安全要点：**

- 原始文件上传后立即解析，解析完成后原始文本在内存中不再保留
- 脱敏预览接口仅返回脱敏后的文本（前 2000 字符），不返回原始文本
- 数据库和对象存储仅持久化脱敏后的文件
- 脱敏映射表（原始位置 → 脱敏类型）仅在内存中用于前端高亮，不持久化
- 临时文件 1 小时后自动清理（Spring 定时任务 + `@Scheduled`）

## 附录 A：状态机代码骨架

```java
@Component
public class ReviewTaskStateMachine {

    @Autowired
    private ReviewTaskRepository taskRepository;

    @Transactional
    public void transition(String taskId, TaskStatus targetStatus) {
        ReviewTask task = taskRepository.findByTaskId(taskId)
            .orElseThrow(() -> new TaskNotFoundException(taskId));

        if (!task.getStatus().canTransitionTo(targetStatus)) {
            throw new IllegalStateException(
                String.format("Cannot transition from %s to %s",
                    task.getStatus(), targetStatus));
        }

        task.setStatus(targetStatus);
        taskRepository.save(task);
    }
}
```
